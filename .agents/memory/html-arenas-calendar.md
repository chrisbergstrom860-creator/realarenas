# Calendar feature (Month / Week / Agenda; insights panel REMOVED)

- Three views: month grid, WEEK (stacked Mon→Sun list reusing the agenda item renderers; empty days render `.wk-rest` "Rest day — nothing scheduled"; whole day-card click opens the shared day panel), agenda. Toggle persists `localStorage arenas_calendar_view`; defaults: desktop month, mobile ≤768px WEEK. **Why week on mobile:** bounded Mon–Sun daily driver; the app thinks in Monday weeks everywhere (leaderboards, stats, goals); agenda stays one tap away.
- The "At a glance" insights panel is GONE (computeInsights/renderInsights/jumpToNextNr, `#cal-insights`, `.cal-layout`, all ins-*/cal-flash CSS deleted). `.cal-shell` max-width 1040px centered = old 1340 − 280 aside − 20 gap, so the grid keeps its cell size. Re-derive shell width before ever adding a side column back.
- Hash scheme: month `#YYYY-MM`; week `#YYYY-MM-DD` = the week's MONDAY. Date-anchored on purpose — ISO week numbers (`#2026-W30`) rejected for ISO-year edge cases. Any valid date deep-links to its week, normalized to Monday form; a week hash forces week view WITHOUT writing the stored preference.
- **Hash canonicalisation must use history.replaceState, never assign location.hash.** Assignment pushes a history entry; normalizing a deep link by push traps browser Back in a bounce loop (raw→canonical→raw…). `syncHash(replace)` contract: replace=true on normalization paths (initial load, hashchange arrivals incl. month-hash-while-week-view, invalid hashes), push only for genuine in-app navigation (prev/next/today/toggle) so Back walks the trail. Invalid hashes (`#2026-02-31`, garbage) recover via `syncHash(true)` — never early-return leaving a stale hash. Nuance: a programmatic/anchor hash assignment itself legitimately pushes one entry — canonicalization must simply not add a SECOND one.
- Week fetch rule: fetched month = month containing the week's MONDAY (`ensureWeek`/`shiftWeek`). Refetch fires only when the Monday changes month; a week merely ENDING in the next month renders its neighbor-month days from `byDayWide` without a refetch (proven: Monday in month M puts Sunday ≤6 days past M's end, inside the ±8/±7 server window).
- Server ±8-day window CANNOT be narrowed: the grid needs ±1 for local-date bucketing and week view needs the full ±7/±8 for neighbor-month week days. Plans window is ±7 text-date; both windows use local fmtYmd, never toISOString.
- Dual-bucket rule (post-insights): `state.byDay` (trimmed to exact month) drives the grid and the agenda trim; `state.byDayWide` (untrimmed) feeds WEEK VIEW **and the DAY PANEL** (renderDayPanel + planFormHtml read byDayWide so week rows can open neighbor-month days — byDayWide ⊇ byDay, so month/agenda panels unaffected). BOTH rebuilt in `load()` AND `refresh()`.
- Week labels: same-month "20–26 Jul 2026", cross-month "29 Jun – 5 Jul 2026", cross-year shows both years. `goToday()` and hash-less init use `nowLocal()` (not `new Date`) so `?_today` shifts them consistently.
- `?_today=YYYY-MM-DD` is a PERMANENT display-only test hook overriding todayYmd()/nowLocal() (regex-validated, never touches writes). **Why:** cross-month weeks only exist on certain real dates — this makes boundary math deterministically testable.
- `planned_sessions` is a user-provisioned table (no DDL via service role) — every read degrades to empty if missing. Plan dates are plain `YYYY-MM-DD` local-date strings (goals convention), never timestamps. **Why:** timestamps skew days in non-UTC locales; text windows compare lexicographically.
- Gating split (dormant behind `PLAN_GATES_ENABLED`): create + content-edit Pro-gated; status-only PATCH (keys ⊆ {status}), DELETE, reads always free. **Why:** lapsed users must be able to close out existing plans. PATCH gate is an inline check, not middleware (middleware can't see body shape). `activity_id` never client-writable.
- "Log this" handoff: calendar → `/log?plan=<planId>`; `POST /api/activities/create` accepts `plan_id` — looked up BEFORE insert (foreign → 403 no insert; stale → ignored; owned → sets activity_id + status done, response carries planCompleted).
- Day panel interactivity: delegated clicks on `#dp-body` via `data-cal-action`; updates re-fetch month + full re-render. Modal desktop / bottom sheet ≤768px; buttons disable in-flight.
- `window.ARENAS_SPORTS` entries have `label`, NOT `name` — `sportName()` must read `.label` (`.name` → undefined → crash on untitled plans).
- `/api/calendar/month` honesty partition: `myStatus` going/interested/'none'; cancelled RSVP → 'none' on club events, row DROPPED for non-club events; non-club events with no RSVP never appear; "Not responded" renders muted. All month arithmetic is integer year/month math (reports-tab rule).
- Mobile grid keeps DOTS, not pills (380px/7 cols ≈ 47px cells); desktop pills cap 5 then "+N more".
- Testing patterns: temp confirmed users via service-role createUser + cookie login on `/html/auth/login`; gate-flag UI via throwaway `.env` + restart; Playwright agents can only reach the proxied app. Gotchas: `events.location` NOT NULL; `clubs` needs `owner_id` + `handle`.
